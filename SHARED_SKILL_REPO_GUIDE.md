# 공용 스킬 저장소 운영 가이드

이 문서는 스킬 저장소를 하나만 유지하고, 여러 작품 프로젝트에서 공용으로 참조해서 사용하는 경우를 위한 가이드다.

적합한 상황:

- 작품 프로젝트가 여러 개다.
- `skills/`와 `Platform/`은 공용으로 관리하고 싶다.
- 작품별 초안, 설정, 메모는 각 프로젝트에서 따로 관리하고 싶다.

## 1. 기본 개념

구조를 두 층으로 나눈다.

- 공용 스킬 저장소
  - `skills/`
  - `Platform/`
  - 공용 문서
- 작품별 프로젝트
  - `drafts/`
  - 작품 설정 문서
  - 플랫폼 적용 파일
  - 작품별 `README.md`

핵심 원칙:

- 스킬과 플랫폼 문서는 공용
- 초안과 설정은 프로젝트별 분리
- 한 프로젝트는 한 작품 기준으로 운영

## 2. 권장 운영 방식

가장 추천하는 방식은 아래 둘 중 하나다.

### 방식 A. 공용 저장소 + 프로젝트별 복사 적용

공용 저장소는 원본으로 유지하고, 각 작품 프로젝트에는 필요한 플랫폼 파일만 복사해서 적용한다.

장점:

- 가장 단순하다.
- 플랫폼별 적용 방식을 프로젝트마다 독립적으로 조절할 수 있다.
- 도구나 런타임 제약이 적다.

단점:

- 공용 운영 파일을 수정하면 각 프로젝트에 다시 반영해야 한다.

이 방식이 적합한 경우:

- 프로젝트 수가 많지 않다.
- 플랫폼 적용 파일을 프로젝트마다 조금씩 다르게 쓰고 싶다.

### 방식 B. 공용 저장소 + 링크 연결

각 작품 프로젝트에서 공용 저장소의 `skills/` 또는 일부 문서를 심볼릭 링크나 정션으로 연결한다.

장점:

- 공용 스킬 수정이 여러 프로젝트에 바로 반영된다.
- 중복 복사가 줄어든다.

단점:

- 링크를 이해하지 못하는 도구가 있을 수 있다.
- 프로젝트별로 다른 버전의 스킬을 쓰기 어렵다.
- 잘못 수정하면 여러 프로젝트에 동시에 영향이 간다.

이 방식이 적합한 경우:

- 항상 최신 공용 스킬을 같이 쓰고 싶다.
- 운영 환경이 링크를 안정적으로 지원한다.

## 3. 권장 디렉터리 예시

```text
shared-skills-repo/
├─ skills/
├─ Platform/
├─ README.md
├─ NEW_PROJECT_CHECKLIST.md
└─ SHARED_SKILL_REPO_GUIDE.md

project-a/
├─ drafts/
├─ project/
├─ AGENTS.md
└─ README.md

project-b/
├─ drafts/
├─ project/
├─ AGENTS.md
└─ README.md
```

## 4. 프로젝트별로 반드시 분리할 것

공용 저장소에 아래 내용을 넣지 않는 것이 좋다.

- 작품별 초안
- 작품별 인물 설정
- 작품별 세계관 설정
- 작품별 플롯 메모
- 작품별 금지사항

이 항목들은 각 프로젝트에 둔다.

권장 구조:

- `project/characters.md`
- `project/world.md`
- `project/plot.md`
- `project/rules.md`
- `drafts/초고`
- `drafts/개고`
- `drafts/원고`

## 5. 플랫폼 파일 적용 원칙

공용 저장소의 `Platform/`은 배포용 원본으로 본다.

실제 적용은 각 프로젝트에서 한다.

주의:

- `Codex`, `GPT-OSS`, `Claude`, `Antigravity` 적용 파일을 한 프로젝트 루트에 섞어 두지 않는다.
- 한 프로젝트에는 현재 사용하는 플랫폼 기준 파일만 둔다.
- 플랫폼을 바꿀 때는 해당 플랫폼 README의 전환 가이드를 먼저 따른다.

## 6. Windows에서 링크 연결하는 방법

Windows에서는 심볼릭 링크보다 정션이 더 편한 경우가 많다.

예시:

```powershell
$SharedRepo = "D:\\shared-skills-repo"
$ProjectRoot = "D:\\project-a"

New-Item -ItemType Junction -Path "$ProjectRoot\\skills" -Target "$SharedRepo\\skills"
New-Item -ItemType Junction -Path "$ProjectRoot\\Platform" -Target "$SharedRepo\\Platform"
```

이렇게 하면 `Project-A`에서 공용 `skills/`와 `Platform/`을 직접 참조할 수 있다.

주의:

- 링크 삭제 시 원본이 아니라 링크만 지우는지 확인한다.
- 일부 백업 도구나 에디터는 링크를 일반 폴더처럼 처리하지 못할 수 있다.

## 7. Git 기준 권장 방식

Git 운영은 아래 중 하나를 추천한다.

### 선택 1. 공용 저장소는 별도 Git, 작품 프로젝트는 별도 Git

가장 안전한 방식이다.

- 공용 스킬 개선은 공용 저장소에서 커밋
- 작품 원고와 설정은 작품 저장소에서 커밋

장점:

- 변경 범위가 명확하다.
- 작품 정보와 공용 운영 자산이 섞이지 않는다.

### 선택 2. 작품 프로젝트에서 공용 저장소를 서브모듈로 연결

가능은 하지만 운영 난도가 조금 올라간다.

적합한 경우:

- Git 사용에 익숙하다.
- 프로젝트별로 공용 스킬 버전을 고정하고 싶다.

주의:

- 서브모듈은 업데이트 절차를 팀이 이해하고 있어야 한다.
- 익숙하지 않으면 오히려 관리 비용이 커진다.

## 8. 실무 권장 조합

대부분의 경우 아래 조합이 가장 현실적이다.

1. 공용 스킬 저장소를 하나 둔다.
2. 작품마다 별도 프로젝트 저장소를 만든다.
3. 공용 저장소의 `Platform/` 문서를 보고 프로젝트 루트에 필요한 파일만 적용한다.
4. `skills/`는 복사하거나 링크로 연결한다.
5. `drafts/`와 `project/` 문서는 작품 프로젝트 안에서만 관리한다.

## 9. 이런 경우는 피하는 것이 좋다

- 한 저장소 안에서 여러 독립 작품의 `drafts/`를 같이 운영하는 경우
- 공용 저장소에 작품 초안을 넣는 경우
- 여러 플랫폼의 운영 파일을 한 프로젝트 루트에 동시에 두는 경우
- 링크 연결과 수동 복사를 섞어 쓰면서 어떤 파일이 원본인지 모르는 상태

## 10. 추천 판단

처음에는 아래처럼 시작하는 것이 가장 안전하다.

- 공용 저장소 1개
- 작품 저장소 여러 개
- `skills/`는 복사 또는 정션
- 플랫폼 파일은 작품 프로젝트에 적용
- 작품 자료와 초안은 각 프로젝트에서만 관리

프로젝트 수가 적으면 복사 방식이 편하고, 프로젝트 수가 많아지면 링크 방식이 더 유리하다.

## 빠른 요약

스킬 저장소 하나를 여러 프로젝트에서 참조하는 방식은 충분히 가능하다. 다만 공용으로 둘 것은 `skills/`와 `Platform/`까지로 제한하고, 작품 초안과 설정은 반드시 프로젝트별로 분리해야 한다. 운영이 단순한 쪽은 복사 방식이고, 동기화가 쉬운 쪽은 링크 방식이다.
