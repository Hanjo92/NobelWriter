# Project Frame Template

Use this file when you need the reusable top block for longform planning output.

Keep this block short, stable, and always first unless the user explicitly asks for another order.

```text
Format:
Structure mode:
Scale:
Genre / audience:
Genre package:
Package selection reason:
Growth model:
Ending promise:
Primary risk:
```

## Field Rules

- `Format`: long novel, web novel, trilogy, saga, season-based serial, or equivalent
- `Structure mode`: linear arc, braided ensemble, omnibus, rotating-focus serial, or equivalent when structurally relevant
- `Scale`: chapter count, arc count, volume count, or the cleanest available scale marker
- `Genre / audience`: combine genre and likely readership tendency in one line
- `Genre package`: one dominant package, with a secondary package only when hybrid is structurally relevant
- `Package selection reason`: mandatory one-sentence justification tied to chapter-return engine, volume-end promise, or dominant failure mode
- if the user gave an ambiguous alias that resolves into another package, name both the raw label and the resolved package in `Package selection reason`
- if the user directly named a genre and the structure pulls toward another package, keep the mapped package in `Genre package` and place the warning outside `project frame`
- `Growth model`: the main expansion logic for the long run
- `Ending promise`: what kind of fulfillment or breakage the work is aiming toward
- `Primary risk`: the one failure mode most likely to break the project if planning stays weak

## Compression Rule

Do not turn `project frame` into an essay. One line per field is enough.

## Hybrid Rule

If hybrid:

- write `Genre package: dominant + secondary`
- keep `Package selection reason` to one sentence that explains why the dominant one still wins
